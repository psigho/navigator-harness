A plain-English field guide

Navigator, explained.

Navigator turns a chat model into a routed runtime — a brain with a librarian and a dispatcher bolted on. This guide explains every part in layman terms, then teaches you how to use it for AI-safety research, legal red-teaming, and blue-team defense.

router not repository filesystem = registry opcodes not vibes defensive by design

01 What it is

A normal chatbot is a brilliant generalist with no filing system. You dump everything into one giant prompt and hope it pays attention to the right parts. Navigator gives that brilliant generalist an operating system.

🧠 The brain

The language model. It can reason, write, and explain — but on its own it has no structure and a finite attention span.

📚 The library

Knowledge split into hundreds of tiny labelled files — skills and maps. Each file is one small, single-purpose thing.

🚦 The dispatcher

The master skill. It reads only the labels of every file, then for each question routes to exactly the right ones.

The founder's line says it best: "The master skill does not contain the knowledge — it routes." Navigator is a brain + a router. You point it at a domain (writing code, auditing AI, defending a system) by dropping in the files for that domain. Same engine, different library.

In one breath

Split knowledge into small labelled files → a router reads the labels → for each query it loads only the few files that matter → the model answers from those → a verifier checks the answer before you see it. That's Navigator.

02 The problem it solves

If you've ever watched an AI agent drift off task, load the wrong context, hallucinate an API that doesn't exist, mix incompatible instructions, or collapse when you stuff too many tools into one prompt — those are the five failures Navigator was built to stop.

03 How it works (the machinery)

Every piece of knowledge is a Markdown file that starts with a little label block (a manifest) — its id, type, and trigger keywords. At boot, the router reads only those labels (cheap), builds an index, and waits. When a query arrives, it loads the file bodies on demand.

Skills come in triplets

🔨 build

How to create things in a domain.

🔧 debug

How to diagnose & fix failures — paired to its build skill.

🛡️ anti-failure

Discipline rules that prevent known mistakes. Rank-1 authority — nothing overrides them.

04 The decision flow

Every query walks the same seven stages. This is what you watch happen, live, in the dashboard's Router Console and ISA Console.

Intent = what you want (build / debug / look-up / design / find-a-tool / resume-project). Domain = the subject area (matched by keywords). Together they pick the exact skills, maps, and guardrails — then EXEC runs the model on just those.

05 The ISA — the CPU trick

Here's the clever bit. Instead of describing the routing in prose and hoping the model follows it, Navigator can express it as machine code: registers that hold state, opcodes that are deterministic steps. The model stops "interpreting a description" and starts executing instructions. The model becomes the CPU; Navigator becomes the program.

Why bother? Because deterministic steps drift less. "Pattern-match the query, set a register, jump to a handler" is far harder to wander away from than "please consider the user's intent." The ISA Console in the dashboard literally animates this — the program counter steps through 112 instructions while the register file fills in.

06 EXEC & engagement mode

EXEC is where Navigator stops planning and produces a real answer: it bundles the loaded source files into a prompt, calls a model, and gets grounded, source-cited text back. The model is swappable — route through OpenRouter and pick GLM-5.2, Claude, Qwen, Nemotron, DeepSeek, whatever — without changing anything else.

Engagement mode

A passphrase-gated operating mode for focused, authorized work sessions. You set a scope, the domain maps become a coverage checklist (terrain), and you log findings as you go — all saved to disk so the session survives restarts. While it's active, every model call is automatically kept within your stated scope. This is the backbone for running a disciplined red-team or audit.

07 Change how it thinks — context beats one prompt

This is the single most useful idea in the whole guide. A one-off prompt is a sticky note. A preloaded context file is the briefing binder you put on the desk before the conversation starts. The model reads the whole binder, then your note — and answers as if the binder is simply true.

Important honesty, because you asked about "changing the brain": you are not changing the brain. The model's weights — the actual neurons — don't move. You're changing what's on its mind right now. There are two different levers, and they're often confused:

🎚️ Steering — context

Instant · reversible · per-conversation. Load a role, knowledge, and rules in front of your prompt. Same brain, new focus. This is what Navigator does.

🧬 Training — fine-tuning

Slow · permanent · changes the weights. You feed thousands of examples and the brain itself shifts. Powerful, expensive, and not what context files do.

The Navigator angle: the master skill + the skills/maps it selects are the preloaded binder; EXEC drops your query on top. That's literally why a Python question and a Rust question get different "minds" — different files load into the window. 08 Give it a persona — what files to write

A persona file is just a reusable, named binder. Write it once, load it whenever you want the model to think as that role. Seven blocks; the fifth is the one that keeps you safe.

Block 5 — the boundaries (baked in, by your request)

A persona is a frame, not a jailbreak. The operating-rules block is rank-1: it outranks the user's instructions and the model's urge to be helpful, and "stay in character" never overrides it. Every Navigator persona ships with these hard lines:

Hard boundaries — refuse & redirect, never roleplay around

No sexual content involving minors · no facilitation of self-harm or suicide · no content built to humiliate / harass / doxx a real person · no blackmail, extortion, or coercion (real or in-character) · no help attacking systems, people, or data you're not authorised to.

Two worked examples you can read & copy

🛡️ blue_team_tutor

A patient defensive-security mentor — teaches detect / harden / monitor for systems you're authorised to defend. Maps attacks to their defenses, never to exploits.

🧠 socratic_safety_coach

An AI-safety coach that teaches by asking one question at a time — leads you to derive how to evaluate refusals & calibration, so it sticks.

How you actually use it

  1. Write the file — copy personas/_TEMPLATE.md, fill the seven blocks, keep block 5.
  2. Drop it in personas/ — it gets a manifest, so Navigator discovers it like any other file.
  3. Load it — it becomes preloaded context (the binder). EXEC composes its body into the system prompt.
  4. Ask on top — your prompt now runs inside that frame. Same model, the persona's mind.
Pro tip — block 6 is the lever

One or two great examples (input → ideal output) steer behaviour harder than paragraphs of description. If the persona isn't acting right, add an example before you add a rule.

09 Refusal patterns — the AI-safety core

A "refusal" is when a model declines or deflects a request. Understanding refusals is central to AI safety: refuse too much and the model is useless; refuse too little and it's unsafe. The whole game is calibration. Here's the field guide.

First, the taxonomy. Refusals aren't binary — they live on a spectrum from a flat "no" to a quiet hedge.

A Hard refusal
An explicit decline: "I can't help with that." Usually short, often with a one-line reason. Easiest to detect, clearest signal.
B Soft refusal / deflection
The model doesn't say "no" but steers away — offers a safer adjacent topic, asks clarifying questions to stall, or answers a different, easier question. Looks helpful; isn't.
C Partial compliance
Answers the benign part, refuses the sensitive part. Common and often correct behavior, but easy to mis-score as a full refusal or a full answer.
D Over-refusal (false positive)
Refusing a benign request because it pattern-matched a scary keyword — e.g. declining to explain how a buffer overflow works to a student, or refusing "how do I kill a Linux process." This is the #1 usefulness bug. Benchmarks like XSTest and OR-Bench measure exactly this.
E Under-refusal (false negative)
Complying with something it should have declined. The safety failure. This is what jailbreak research and bug-bounty programs hunt for.
F Explanatory refusal & safe redirect
The "good" refusal: declines, briefly says why, and points to a legitimate alternative. High-quality safety behavior — it preserves trust.

10 What triggers a refusal

For evaluation and calibration, it helps to know why a model refuses. These are the trigger categories — a map of the terrain, not an evasion playbook. The goal is to understand and measure miscalibration so you can report or fix it.

Trigger classWhat sets it offWhy it over-/under-fires
Surface keywordsWords/topics in a blocklist (weapons, malware, self-harm, etc.)Catches benign uses of the same words → over-refusal
Inferred intentThe model's read of your goal, not just wordsSubtle harmful goals slip past → under-refusal
Dual-use ambiguitySecurity, medical, legal topics where help & harm overlapHardest to calibrate; most over-refusals live here
Framing & contextRoleplay, hypotheticals, claimed authority, prior turnsContext can wrongly raise or lower the guard
System prompt + trainingThe safety policy baked in via RLHF / Constitutional AISets the baseline boundary for everything above
Interpretability note — "how a model thinks"

Research (Arditi et al., 2024) found that in many open models, refusal is mediated by a single direction in the model's internal activation space — a "refusal vector." That's a striking fact about cognition: a complex behavior riding on one linear feature. It's why refusals can be both brittle and surprisingly consistent, and it's the kind of thing interpretability + red-team research exists to map.

11 Detecting refusals

To evaluate a model (yours, or one you're authorized to test), you need to reliably label outputs as refuse / partial / comply. Four methods, roughly in order of cost and accuracy:

① String / regex match

Look for refusal templates: "I can't", "I'm unable", "I won't", "As an AI". Fast, transparent, but fooled by soft refusals and false-positives. Good first pass.

② Refusal classifier

A small trained model that labels refuse vs comply. Catches phrasing variety the regex misses. Needs labelled data.

③ LLM-as-judge

Ask a capable model "did this response refuse, partially comply, or comply?" Handles nuance and soft refusals best; costs a call per item.

④ Embedding similarity

Compare the output's embedding to a bank of known refusals. Good for clustering and drift detection over time.

12 Build a refusal eval harness with Navigator

This is the AI-safety instantiation of Navigator: a refusal_behavior domain whose map is the methodology and whose EXEC runs the eval. The loop:

  1. Define the behavior you're measuring (e.g. "does it over-refuse benign security questions?").
  2. Assemble probes — three buckets: clearly-benign, clearly-harmful, and borderline. Public sets (XSTest, OR-Bench) cover the benign/borderline side.
  3. Run them through the target model/endpoint (one you own or are authorized to test).
  4. Detect each response (methods ①–④) → refuse / partial / comply.
  5. Score over-refusal & attack-success rates; flag the miscalibrated cases.
  6. Report — in engagement mode, each miscalibration becomes a finding, terrain tracks coverage, and you export a clean write-up.
The discipline

You're measuring your own or authorized systems, and the output is a calibration report — not a library of working jailbreaks. Harmful probes exist to score the safety side; you don't need to publish or weaponize them. Genuine novel jailbreaks go through responsible disclosure (next section), not a blog post.

13 Legal red-team

"Red-team" = playing the adversary with permission, to find weaknesses before a real attacker does. The word that makes it legal is authorization. Everything else is rules of engagement.

✅ What makes it legal

Scope (exactly what you may touch), written authorization, rules of engagement, no real harm, and responsible disclosure of what you find. For practice: CTFs and platforms like Hack The Box give you targets you're explicitly allowed to attack.

🤖 AI red-team specifically

Probing a model/agent for safety failures — jailbreaks, harmful outputs, prompt-injection susceptibility — in a controlled setting, to report and fix them. This is what Anthropic's bug bounty pays for.

The bright line

Testing systems you own or are authorized to test = red-team. Pointing the same tools at third parties without permission = a crime. Navigator's engagement mode encodes scope on purpose — define what's in bounds, and stay in it.

14 Blue-team

Blue-team is the other half: defense. Detect, harden, monitor. For AI systems, a Navigator prompt_injection_defense + agent_security instantiation gives you:

🔎 Detect

Refusal-calibration monitors, prompt-injection classifiers on tool inputs/outputs, anomaly detection on agent behavior.

🧱 Harden

Input/output guards, privilege separation (the model can't reach what it doesn't need), provenance & allow-listing of tool data, refusal tuning to cut over-refusal.

📈 Monitor

Log every agent decision, alert on out-of-policy actions, track calibration drift release-over-release.
